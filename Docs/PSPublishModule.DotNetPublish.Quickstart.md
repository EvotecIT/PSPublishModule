# DotNet Publish Quickstart (PSPublishModule / PowerForge)

This quickstart is the practical entrypoint for the DotNet publish engine.

## Naming Convention (Use This Consistently)

- `New-DotNetPublishConfig`
  - scaffolds a JSON config file on disk.
- `Invoke-DotNetPublish`
  - validates, plans, runs, or exports config.
- `New-ConfigurationDotNet*`
  - creates DSL objects used inside `-Settings { ... }`.

This keeps responsibilities clear and avoids command mix-ups.

## Recommended Flow: JSON First

1. Scaffold config:

```powershell
New-DotNetPublishConfig -ProjectRoot '.' -PassThru
```

2. Open and edit `powerforge.dotnetpublish.json`.

3. Validate plan only:

```powershell
Invoke-DotNetPublish -ConfigPath '.\powerforge.dotnetpublish.json' -Validate
Invoke-DotNetPublish -ConfigPath '.\powerforge.dotnetpublish.json' -Plan
```

4. Run publish:

```powershell
Invoke-DotNetPublish -ConfigPath '.\powerforge.dotnetpublish.json' -ExitCode
```

## Fast Run Overrides (No File Edits Needed)

Use overrides for local/CI experiments while keeping base JSON stable:

```powershell
Invoke-DotNetPublish `
  -ConfigPath '.\powerforge.dotnetpublish.json' `
  -Target 'service' `
  -Runtimes 'win-x64' `
  -Frameworks 'net10.0-windows' `
  -Styles PortableCompat `
  -SkipRestore `
  -SkipBuild `
  -ExitCode
```

## DSL Flow (When You Want Scripted Composition)

```powershell
Invoke-DotNetPublish -Plan -Settings {
    New-ConfigurationDotNetPublish -IncludeSchema -ProjectRoot '.' -Configuration 'Release'

    New-ConfigurationDotNetProject -Id 'service' -Path 'src/Service/Service.csproj'
    New-ConfigurationDotNetTarget -Name 'service' -ProjectId 'service' -Framework 'net10.0-windows' -Runtimes 'win-x64' -Style PortableCompat -Zip
}
```

Use this when config composition is dynamic or generated by build scripts.

## CLI Equivalents

- Scaffold:

```powershell
powerforge dotnet scaffold --project-root . --output json
```

- Publish:

```powershell
powerforge dotnet publish --config .\powerforge.dotnetpublish.json --plan
powerforge dotnet publish --config .\powerforge.dotnetpublish.json --output json
```

## What the Engine Produces

Depending on config, the run can emit:

- published output folders
- zip artifacts
- service scripts (`Install-Service.ps1`, `Uninstall-Service.ps1`, `Run-Once.ps1`)
- MSI prepare/build outputs
- benchmark gate results
- manifests and checksums (for example `SHA256SUMS.txt`)

## Migration Rule of Thumb

For TestimoX/TierBridge/Sectigo-style repos:

1. start by expressing publish/sign/package steps in JSON,
2. then add service/MSI/state primitives,
3. only replace legacy script logic after parity is validated.
